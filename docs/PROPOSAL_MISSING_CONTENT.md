## Executive Summary Project (Max. 150 Words)
Our project proposes an AI-Based Intermodal Transit Synchronization System for Jakarta's public transportation network. The system supports the Hackathon theme by using AI, data integration, and real-time analytics to improve public mobility. The main problem addressed is the uncertainty of intermodal transfer time caused by disconnected schedules, fluctuating traffic, passenger crowding, walking distance between transit points, and operational disruptions. The solution predicts vehicle arrival time, estimates passenger transfer time, monitors station or stop density, and recommends operational actions such as dwell-time adjustment, safe speed adjustment, or directing passengers to the next less-crowded vehicle. The prototype integrates a FastAPI backend, a Streamlit dashboard, synthetic tap-in/tap-out data, route schedules, transfer walking time, incident reports, and an ETA optimization engine. For Dishub DKI Jakarta, the system can improve service reliability, reduce passenger waiting time, and support the shift from private vehicles to public transport.

## Main Objectives (Max. 200 Words)
The main objective is to build an AI-supported decision system that improves intermodal connectivity and reduces passenger waiting time during public transport transfers in Jakarta. The first target is to estimate vehicle arrival time more accurately by combining schedule data, vehicle position, traffic delay, and disruption information. The second target is to optimize transfer connections between rail-based modes such as MRT, LRT, and KRL and road-based modes such as Transjakarta, Mikrotrans, school buses, and feeder services. The third target is to keep intermodal waiting time below 8 minutes at priority transfer nodes by recommending dwell-time adjustment, safe operational speed adjustment, or alternative vehicle options. The fourth target is to monitor passenger density using tap-in/tap-out data or synthetic/anonymized samples for prototype purposes, so passenger distribution can be improved. The fifth target is to provide a dashboard for operators and integrated travel information for users through the existing JakLingko ecosystem. The final target is to prepare a scalable prototype that can be expanded into a pilot project with real operator data.

## 4.1 What Makes Your Solution Different? (Max. 200 Words)
The main difference of this solution is that it does not only display arrival information, but also provides a closed-loop operational recommendation. Existing transport applications usually inform users about routes, estimated arrival, or fare, while this system connects passenger demand, vehicle supply, transfer walking time, crowding, and disruption data into one decision engine. The system treats intermodal transfer as an operational problem, not only as a navigation problem. It calculates when passengers will arrive at the next mode, whether the next vehicle should wait, whether passengers should be directed to a less-crowded vehicle, and whether road-based vehicles can safely adjust their speed to improve the connection. Another unique element is the focus on non-rail adjustment because rail-based services such as MRT, LRT, and KRL have stricter schedules. The prototype also includes driver incident reporting, synthetic tap-in/tap-out density simulation, and a dashboard that can be used by operators. This makes the solution practical for both passengers and transport authorities.

## 4.2 Positioning Compared to Existing Approaches or Products (Max. 200 Words)
This solution is positioned as an enhancement to existing mobility platforms, especially JakLingko, rather than a replacement. Current public transport applications generally focus on journey planning, fare integration, route information, and static or semi-real-time arrival information. Our system complements those functions by adding an operational intelligence layer that synchronizes intermodal connections. Compared to standard ETA applications, this solution considers transfer walking time, passenger density, dwell-time feasibility, incident reports, and the operational constraint that rail-based schedules should not be interrupted. Compared to conventional operator dashboards, this system does not only monitor performance but also recommends actions, such as holding a feeder vehicle within a safe dwell-time limit, directing passengers to the next less-crowded vehicle, or adjusting non-rail vehicle speed within safety rules. Therefore, the solution improves existing approaches by making the data actionable. It can be integrated through APIs into JakLingko, Dishub dashboards, or operator control centers, allowing gradual adoption without rebuilding the entire transport ecosystem.

## 5.1 Main Solution - Key Technologies (Max. 150 Words)
The solution uses AI, data analytics, API integration, web dashboard, and real-time decision support. The key technologies are: Python for data processing and modeling; FastAPI for backend services; Streamlit for the operator dashboard; Pandas for schedule, tap-in/tap-out, and fleet data processing; Scikit-learn for ETA delay prediction; OpenStreetMap or manually curated transfer nodes for walking-time estimation; and REST API endpoints for integration with JakLingko or operator systems. The prototype also uses synthetic tap-in/tap-out data, schedule data, transfer node data, incident reports, and fleet capacity assumptions. In production, the same architecture can connect to AFC transaction logs, vehicle GPS/AVL, GTFS or GTFS-Realtime, traffic feeds, weather/flood APIs, and operator incident logs.

## 5.2 Technology Selection and Implementation (Max. 150 Words)
Python was selected because it is fast for prototyping AI, data pipelines, and transport simulations. FastAPI was selected because it can expose ETA, density, optimization, and incident modules as lightweight APIs that are easy to connect with mobile apps or dashboards. Streamlit was selected for rapid dashboard development, allowing judges and operators to test transfer scenarios without complex frontend development. Pandas is used to process schedule, stop, transfer, density, and fleet data. Scikit-learn is used for the first machine learning model because it is explainable, lightweight, and suitable for tabular operational data. The implementation starts with public and synthetic data, then replaces each data source with real operator feeds during pilot deployment. This approach allows the prototype to work immediately while keeping the architecture ready for production integration.

## 5.3 Solution Algorithm (Max. 150 Words)
The algorithm has four layers. First, the ETA module predicts the arrival time of each vehicle using schedule data, delay features, and optional GPS or traffic inputs. Second, the transfer module adds walking time from the first stop or station to the next transfer point. Third, the optimization module compares passenger arrival time with the next vehicle departure time and applies an 8-minute waiting-time threshold. If the connection is feasible, the system recommends normal transfer. If the next vehicle departs too early and dwell-time extension is still safe, the system recommends holding the non-rail vehicle. If the vehicle is crowded or the connection is not feasible, the system recommends the next vehicle or alternative route. Fourth, the incident module updates ETA and recommendations when driver reports, flood, traffic, or operational disruptions occur.

## 5.4 Primary Data or Input Used (Max. 200 Words)
The prototype uses a combination of schedule data, stop data, transfer nodes, synthetic tap-in/tap-out data, fleet capacity assumptions, and incident reports. Schedule and stop data represent the baseline network and arrival/departure times. Transfer node data contains walking-time estimates between transport points, such as a Transjakarta stop to an MRT or KRL station. Synthetic tap-in/tap-out data is used to simulate passenger density while protecting privacy and avoiding dependence on restricted transaction data during the prototype stage. Fleet capacity data is used to estimate load factor and crowding-related recommendations. Incident data is used to simulate disruptions such as flooding, congestion, accidents, or operational delays. The data quality is sufficient for proof-of-concept because it demonstrates the logic of ETA prediction, density estimation, and intermodal optimization. For production, the system should use official operator data, including AFC tap-in/tap-out, AVL/GPS, actual trip logs, GTFS/GTFS-Realtime, fleet registry, service alerts, and incident logs.

## 5.5 Security and Scalability Considerations (Max. 150 Words)
The system follows a privacy-by-design approach. It does not require raw personal identity data. Passenger transaction data should use hashed and salted card or user IDs, while public dashboards should only show aggregated density and performance indicators. Access to raw operational data must follow role-based permissions, audit logging, encryption in transit, and secure data retention policies. The architecture is scalable because each function is separated into modules: ETA prediction, density estimation, transfer optimization, incident reporting, and dashboard visualization. These modules can be deployed as APIs and expanded gradually from a small pilot corridor to citywide operation. The system can also support future integration with GTFS-Realtime, live GPS, traffic APIs, weather and flood alerts, and operator command centers without changing the core decision logic.

## 6.3 Development Stages (Max. 180 Words)
Stage 1 is data preparation: collect public schedule data, stop data, route data, transfer nodes, walking-time assumptions, fleet capacity assumptions, and synthetic tap-in/tap-out samples. Stage 2 is MVP development: build the ETA baseline, density estimator, transfer optimizer, dwell-time recommendation, safe speed adjustment, incident reporting, FastAPI backend, and dashboard. Stage 3 is prototype testing: simulate several priority transfer nodes such as Dukuh Atas, Tanah Abang, Manggarai, Harmoni, Blok M, and Lebak Bulus. Stage 5 is pilot deployment: integrate the system with JakLingko or operator control dashboards and evaluate KPIs such as ETA accuracy, average waiting time, transfer miss rate, load factor balance, and user satisfaction.
